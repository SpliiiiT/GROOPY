Section 00

Big picture — what is GROOPY?

GROOPY is a bidirectional sign-language communication proof-of-concept, between a Deaf/HoH person (the signer) and a hearing non-signer. Two directions plus one shared layer, all in a single repo.

TrackFolderDirectionWhat powers it
A — Recognitionrecognition/Sign → Text/SpeechCNN (letters) + sequence model (words)
B — Synthesissynthesis/Text/Speech → SignASR → gloss → sign plan → clip playback
shared/shared/source of truthcontract.py, vocabulary.py, config.py
sentiment/sentiment/tone analysis3-model bake-off
The core idea — "bake-off"

For each task we don't train one model — we put several in competition under an identical protocol, then a weighted scorecard picks the winner. That's what shows we understood the trade-off (accuracy / speed / size), not just that we can call .fit().

Methodology: CRISP-DM (Business understanding → Data → Modeling → Evaluation → Deployment). Every bake-off follows these steps.

1.2  The Recognition pipeline (Sign → Text), live

Fingerspelling (letters) · desktop/app.py

  • Webcam frame → cropped to 224×224, normalized to [0,1].
  • Hand-presence gate — no hand visible, no prediction.
  • CNN outputs a softmax over 29 classes (A–Z + del, nothing, space).
  • Confidence gate (CONFIDENCE_GATE = 0.80): emit a token only if probability ≥ 0.80.
  • Debounce (DEBOUNCE_MS = 500): at most one prediction per 500 ms.
  • Capture-to-commit: a letter is added only on a deliberate action — no 30-letters/second spew.

Word signs · word_stream.py

  • MediaPipe Holistic extracts landmarks per frame: pose 33 + left hand 21 + right hand 21.
  • Each frame → 258 features = pose 33×(x,y,z,vis) + LH 21×(x,y,z) + RH 21×(x,y,z).
  • Stack 30 frames (SEQ_LEN=30) → a sequence (30, 258).
  • Sequence model outputs a softmax over 20 classes.
The decisive fix — landmark normalization

holistic.normalize_sequence recenters x,y on the shoulder midpoint and divides by shoulder width (pose 11/12). This single change took accuracy from ~5% (chance) → 37% → 70%. Applied identically in training and live, so there is no "train/serve skew".

1.3  The shared layer (the "contract")

shared/vocabulary.py defines 20 words (hello, thanks, please, yes, no, sorry, help, want, need, name, you, me, good, bad, happy, sad, eat, drink, friend, love). This list is simultaneously the word model's output classes and the synthesis clip keys — one place to change, both directions update. Any word outside it is handled by fingerspelling, so the system never dead-ends.

1.4  Reproducibility

SEED = 42 everywhere, same splits, same hyperparameters for every candidate of a bake-off — that's what makes the comparison fair.

Section 02 · Axis 2

Model architectures — layers & hyperparameters

2.A  Fingerspelling CNN

Task: classify a hand image into one of 29 classes. Data: Kaggle ASL Alphabet, 87,000 images, 3000/class. Input: 224×224×3, [0,1].

Candidate 1 — cnn_scratch (the "built from scratch" baseline)

The "we actually understood CNNs" model — every layer is ours. Repeated VGG-style block: [Conv2D → BatchNorm → ReLU] ×2 → MaxPooling2D.

StageLayersFilters
Block 1Conv-BN-ReLU ×2 + MaxPool32
Block 2Conv-BN-ReLU ×2 + MaxPool64
Block 3Conv-BN-ReLU ×2 + MaxPool128
Block 4Conv-BN-ReLU ×2 + MaxPool256 ← Grad-CAM target
HeadGlobalAveragePooling2D → Dropout(0.4) → Dense(256, ReLU) → Dropout(0.3) → Dense(29, softmax)
  • Conv2D: 3×3 kernels, padding="same", use_bias=False — the following BatchNorm cancels any bias, so it's a wasted parameter.
  • GlobalAveragePooling instead of Flatten: far fewer params (small model, ~1.25M / ~15 MB) and clean Grad-CAM.
  • Dropout 0.4 / 0.3: regularization against overfitting. ~1,248,381 parameters.

Candidates 2–4 — transfer learning (ImageNet backbones)

Same head for all: GAP → Dropout(0.3) → Dense(29, softmax).

Two-phase fine-tuning (pretrained only)

Phase 1 — backbone frozen, train only the new head (LR 1e-3). Phase 2 — unfreeze the last ~30 backbone layers, keep BatchNorm frozen (standard practice), retrain at LR/10. The cnn_scratch trains in a single phase (nothing to unfreeze).

2.B  Sequence model for word signs

Task: classify a sequence (30, 258) into one of 20 words. A signed word is a motion, so candidates are sequence models, not CNNs. All share Input(30,258) → Masking(0.0) (ignores padding frames) → … → Dense(20, softmax).

bilstmBi-LSTM(96, seq) → Drop → Bi-LSTM(48) → Drop → Dense(64)372,692reads the sign both ways
transformerDense(128)+pos-emb → 2× attention block → GAP1D → Dense(64)241,876attention, parallelizable

The Transformer block (pre-norm, residuals)

LayerNorm → MultiHeadAttention(4 heads, key_dim=32) → + residual
LayerNorm → Dense(128, ReLU) → Dropout → Dense(d_model) → + residual

Learned positional embedding (Embedding(30, 128)) tells the model which frame it is — attention alone is order-invariant. d_model = 128, 2 stacked blocks, GlobalAveragePooling1D before the head.

Section 03 · Axis 3

Evaluation metrics

3.1  The principle: weighted scorecard (scorecard.py)

We do not judge on accuracy alone. Each metric is min-max normalized to [0,1] across candidates ("lower is better" metrics like latency are inverted), then weighted. The total picks the winner.

Bake-offWeights
CNNaccuracy 40% · latency 20% · size 15% · robustness 15% · stability 10%
Wordsaccuracy 60% · latency 20% · size 20% (100% automatic)
Sentimentaccuracy 50% · latency 30% · size 20%

robustness & stability are manual 0–1 scores — the human check "the model looks at the hand and stays stable live" is part of the decision.

The story to tell

EfficientNet wins on accuracy but is the slowest. cnn_scratch is the one you'd actually deploy in real time (26 ms, 15 MB, 99.3%). "The scorecard encodes the priorities" — give latency more weight and the in-house baseline wins. Exactly the nuance the professor wants to hear.

Nuance to know

The scorecard picks gru, but we keep the transformer as the live default — within noise of gru (74.6% vs 75.4%) and ~7× faster (21 vs 142 ms), which matters more for capture-to-commit UX. bilstm has the best raw accuracy (77.97%) but its latency/size sink its score. The best story in the whole project

The IMDB-accuracy scorecard picks distilbert. But IMDB has no neutral examples, so the metric can't test neutral detection. On a realistic test (eval_realistic.py), distilbert misses all neutrals (only trained to force pos/neg) while twitter_roberta scores 100%. So RECOMMENDED_MODEL = "twitter_roberta". The wrong benchmark picks the wrong model — a real data-science lesson, not just a number.

Section 04 · Axis 4

XAI — explainability

4.1  Grad-CAM on the CNN (xai_gradcam.py)

Grad-CAM (Gradient-weighted Class Activation Mapping) answers: which regions of the image does the CNN rely on? It produces a heatmap overlaid on the image.

How it works — be able to explain

  • Take the last convolutional layer (for cnn_scratch, the 256-filter block) — its activation maps keep spatial resolution.
  • Compute the gradient of the predicted class w.r.t. those maps (tf.GradientTape).
  • Global-average the gradients → one importance weight per map.
  • Weighted combination of maps → ReLU (keep what pushes toward the class) → normalize to [0,1] → resize to 224×224 and overlay in jet.

What we found (recognition/results/gradcam_*.png)

  • EfficientNetB0 focuses tightly on the hand → robust.
  • cnn_scratch sometimes drifts onto the background on hard cases → less robust.
  • Hard pairs M/A, M/E, Q/G (predicted during EDA) are confirmed → this closes the CRISP-DM loop: exploration hypothesis verified by explainability.
  • These observations feed the "robustness" score; after re-scoring, EfficientNet wins even wider (0.681).
Why it matters here

In sign language, a model that "cheats" on background or skin tone would be biased and unfair. Grad-CAM is therefore a trust and bias check, not just a pretty picture.

4.2  Interpretability of the other models

  • Sequence model (words): no Grad-CAM (no image), but explainability comes from landmark normalization (the model sees invariant hand/body geometry, not raw pixels) and Masking (we know exactly which frames count). The confusion matrix shows which words get mixed up.
  • Sentiment: the scratch model (TF-IDF + LogReg) is intrinsically interpretable — you can read the word weights. The neutral-band analysis (§3.4) itself explains the model's behavior.
Section 05

Cheat sheet — likely questions

QuestionShort answer
CNN for letters, RNN/Transformer for words — why?A letter is a static image (CNN); a word is a motion over time (sequence).
Why use_bias=False in the Convs?The following BatchNorm cancels any bias → useless parameter.
GlobalAveragePooling vs Flatten?Far fewer params (small model) and clean Grad-CAM.
Role of Dropout?Regularization, prevents overfitting — crucial with little data.
What unlocked the word model?Landmark normalization (5%→37%→70%) + WLASL/ASL Citizen merge (791 sequences).
Why two-phase fine-tuning?Protect ImageNet weights: train head first, then unfreeze gently (LR/10, BatchNorm frozen).
What is the scorecard?A multi-criteria decision (accuracy + latency + size + robustness), normalized and weighted — not just accuracy.
Best example of evaluation rigor?Sentiment: IMDB picks distilbert, but it misses all neutrals; twitter_roberta wins the realistic test.
What is Grad-CAM for here?A heatmap of the regions driving the prediction → verify the model looks at the hand (trust + anti-bias).
How do you control overfitting?EarlyStopping (restore_best_weights), Dropout, augmentation, ReduceLROnPlateau, 15% validation split.
Section 06

How to run it (demo)

# Fingerspelling CNN — full bake-off
python -m recognition.src.train --model all
python -m recognition.src.evaluate            # metrics + scorecard
python -m recognition.src.xai_gradcam --model recognition/models/cnn_scratch.keras --n 12

# Words (sequence)
python -m recognition.src.train_word --model all --epochs 60 --augment 8
python -m recognition.src.evaluate_word

# Sentiment
python -m sentiment.src.train_scratch
python -m sentiment.src.evaluate
python -m sentiment.src.eval_realistic

# Desktop demos (live webcam)
python desktop/app.py             # fingerspelling
python desktop/synthesis_app.py   # text → signs

# Tests
pytest tests/test_smoke.py

Environment: local .venv, TensorFlow 2.17.1 / Keras 3.15.0 (pinned in requirements.txt). Pretrained sentiment models run on PyTorch CPU (sentiment/requirements.txt).
